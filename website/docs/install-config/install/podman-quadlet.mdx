This installation method is for test setups and small-scale production setups that run authentik with [Quadlet](https://docs.podman.io/en/latest/markdown/podman-systemd.unit.5.html)-managed Podman containers.

Quadlet turns Podman `.container`, `.pod`, and `.volume` files into systemd services. Use this installation method when you want Podman with native systemd lifecycle management instead of `podman compose`. For larger production deployments, see [Kubernetes installation](./kubernetes.md).

## Requirements

- A host with at least 2 CPU cores and 2 GB of RAM
- Podman 5.0 or later
- systemd

To check the installed Podman version, run:

```shell
podman --version
```

## Rootful vs. rootless

| Aspect                          | Rootful                        | Rootless                                               |
| ------------------------------- | ------------------------------ | ------------------------------------------------------ |
| Unit directory                  | `/etc/containers/systemd/`     | `~/.config/containers/systemd/`                        |
| Reload command                  | `sudo systemctl daemon-reload` | `systemctl --user daemon-reload`                       |
| Service manager                 | system service manager         | user service manager                                   |
| Podman API socket path          | `/run/podman/podman.sock`      | `$XDG_RUNTIME_DIR/podman/podman.sock`                  |
| Support for Outpost auto-deploy | Yes                            | Yes, with `loginctl enable-linger`                     |
| Binding ports below 1024        | Yes                            | Requires sysctl, firewall, socket activation, or an LB |

Every command in this guide is shown for both modes; pick the tab that matches your deployment.

## Prepare the host

Create the directories that the containers use for state and configuration. Then create an environment file with a PostgreSQL password and an authentik secret key.

<Tabs groupId="install-mode">
<TabItem value="rootful" label="Rootful" default>

```shell
sudo mkdir -p /var/lib/authentik/{data,certs,custom-templates}
sudo mkdir -p /etc/authentik
sudo tee /etc/authentik/authentik.env > /dev/null <<EOF
POSTGRES_PASSWORD=$(openssl rand -base64 36 | tr -d '\n')
AUTHENTIK_SECRET_KEY=$(openssl rand -base64 60 | tr -d '\n')
AUTHENTIK_ERROR_REPORTING__ENABLED=true
EOF
sudo chmod 600 /etc/authentik/authentik.env
```

</TabItem>
<TabItem value="rootless" label="Rootless">

```shell
mkdir -p ~/.local/share/authentik/{data,certs,custom-templates}
mkdir -p ~/.config/authentik
tee ~/.config/authentik/authentik.env > /dev/null <<EOF
POSTGRES_PASSWORD=$(openssl rand -base64 36 | tr -d '\n')
AUTHENTIK_SECRET_KEY=$(openssl rand -base64 60 | tr -d '\n')
AUTHENTIK_ERROR_REPORTING__ENABLED=true
EOF
chmod 600 ~/.config/authentik/authentik.env
```

</TabItem>
</Tabs>

Enable the Podman API socket so that the authentik worker can deploy and manage outposts:

<Tabs groupId="install-mode">
<TabItem value="rootful" label="Rootful" default>

```shell
sudo systemctl enable --now podman.socket
```

</TabItem>
<TabItem value="rootless" label="Rootless">

```shell
systemctl --user enable --now podman.socket
loginctl enable-linger "$USER"
```

</TabItem>
</Tabs>

## Download the Quadlet units

The deployment uses one pod for PostgreSQL, the server, and the worker. Because they share a pod network namespace, the containers can reach each other on `localhost` without a custom network.

The unit files are committed under [`lifecycle/quadlet/`](https://github.com/goauthentik/authentik/tree/main/lifecycle/quadlet) in the authentik repository and published to `docs.goauthentik.io/quadlet/`. The `Image=` line on the server and worker units is pinned to the same authentik tag as the [Docker Compose file](./docker-compose.mdx). To pin a different tag, edit the `Image=` lines after downloading.

The rootless variants live under `lifecycle/quadlet/rootless/`, and under `/quadlet/rootless/` on the documentation site. They use the systemd specifiers `%h` for the home directory and `%t` for `$XDG_RUNTIME_DIR`, so the same files work for any rootless account.

<Tabs groupId="install-mode">
<TabItem value="rootful" label="Rootful" default>

```shell
sudo mkdir -p /etc/containers/systemd
for unit in \
    authentik.pod \
    authentik-database.volume \
    authentik-postgresql.container \
    authentik-server.container \
    authentik-worker.container; do
    curl -fsSL "https://docs.goauthentik.io/quadlet/$unit" \
        | sudo tee "/etc/containers/systemd/$unit" >/dev/null
done
```

</TabItem>
<TabItem value="rootless" label="Rootless">

```shell
mkdir -p ~/.config/containers/systemd
for unit in \
    authentik.pod \
    authentik-database.volume \
    authentik-postgresql.container \
    authentik-server.container \
    authentik-worker.container; do
    curl -fsSL "https://docs.goauthentik.io/quadlet/rootless/$unit" \
        -o "$HOME/.config/containers/systemd/$unit"
done
```

</TabItem>
</Tabs>

The worker shares the pod with the server. Because both default to listening on `:9000` for HTTP and `:9300` for metrics, the worker unit shifts its listeners to `:9001` and `:9301` so the listeners do not collide inside the pod's network namespace. The worker also mounts the Podman API socket so that it can deploy outposts.

`User=0:0` on the worker runs the process as UID 0 inside the container so that it can read the bind-mounted Podman socket. Under rootful Podman, the host socket is root-owned. Under rootless Podman, UID 0 inside the container maps back to the calling user through the user namespace.

## Start authentik

Reload systemd so that Quadlet generates the service units. Then enable and start the generated container services. The pod starts automatically as a dependency.

<Tabs groupId="install-mode">
<TabItem value="rootful" label="Rootful" default>

```shell
sudo systemctl daemon-reload
sudo systemctl enable --now \
    authentik-postgresql.service \
    authentik-server.service \
    authentik-worker.service
```

</TabItem>
<TabItem value="rootless" label="Rootless">

```shell
systemctl --user daemon-reload
systemctl --user enable --now \
    authentik-postgresql.service \
    authentik-server.service \
    authentik-worker.service
```

</TabItem>
</Tabs>

Quadlet pulls images on first start, so the initial run can take a few minutes. Tail the logs across all three containers to watch the migration and confirm that each service starts successfully:

<Tabs groupId="install-mode">
<TabItem value="rootful" label="Rootful" default>

```shell
journalctl -u authentik-server.service -u authentik-worker.service -u authentik-postgresql.service -f
```

</TabItem>
<TabItem value="rootless" label="Rootless">

```shell
journalctl --user -u authentik-server.service -u authentik-worker.service -u authentik-postgresql.service -f
```

</TabItem>
</Tabs>

## Access authentik

To start the initial setup, navigate to `http://<your server's IP or hostname>:9000`.

You are then prompted to set a password for the `akadmin` user (the default user).

## Privileged ports (below 1024)

The pod's `PublishPort=` lines bind on the host's network namespace, so any option that lets a rootless process reach ports 80 and 443 works here. Common options include:

- **Reverse proxy or load balancer**: terminate TLS at nginx, Caddy, Traefik, or HAProxy and forward to authentik on the high ports.
- **Firewall port-forwarding**: redirect `80` to `9000` and `443` to `9443` at the host firewall (`iptables`, `nftables`, or `firewalld`).
- **systemd socket activation**: have systemd bind the privileged socket and pass it into the pod.
- **Lower the unprivileged port floor**: `sudo sysctl net.ipv4.ip_unprivileged_port_start=80`, persisted in `/etc/sysctl.d/`. This relaxes the system-wide guard rail; see the [Rootless Privileged Ports discussion](https://tangentsoft.com/podman/wiki?name=Rootless+Privileged+Ports&p) for the trade-offs.

## Upgrading

The Quadlet unit files set `AutoUpdate=registry` on each `.container` file. Podman's built-in updater can pull newer digests for the configured image tag and restart containers. Enable the timer:

<Tabs groupId="install-mode">
<TabItem value="rootful" label="Rootful" default>

```shell
sudo systemctl enable --now podman-auto-update.timer
```

</TabItem>
<TabItem value="rootless" label="Rootless">

```shell
systemctl --user enable --now podman-auto-update.timer
```

</TabItem>
</Tabs>

To upgrade on demand:

<Tabs groupId="install-mode">
<TabItem value="rootful" label="Rootful" default>

```shell
sudo podman auto-update
```

</TabItem>
<TabItem value="rootless" label="Rootless">

```shell
podman auto-update
```

</TabItem>
</Tabs>

To upgrade to a new authentik release, download the current Quadlet units again or edit the `Image=` line on `authentik-server.container` and `authentik-worker.container` to the desired tag. Then reload systemd and restart the container services.

## SELinux

On SELinux-enabled distributions such as RHEL, Rocky Linux, and Fedora, every `Volume=` line uses the `:Z` suffix. This relabels the host path with a private container label.

The Podman socket bind mount also uses `:Z`. If you would rather not relabel the system socket, use `:z` (shared label) or drop the suffix and add a permissive policy explicitly.

## Use a Podman socket on earlier authentik versions

In versions before 2026.8, authentik's startup task only auto-detected `/var/run/docker.sock`. To use a Podman socket on those versions, bind-mount it to the Docker path inside the worker:

```ini
Volume=/run/podman/podman.sock:/var/run/docker.sock:Z
```

Alternatively, set `DOCKER_HOST=unix:///run/podman/podman.sock` in `/etc/authentik/authentik.env`; the Docker SDK that authentik uses honors this variable.
