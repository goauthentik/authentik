Administrators for an authentik instance work primarily in the Admin interface; that is where administrators create new users and groups, add applications, and manage system settings.

End-users, who are accessing their applications via authentik for their daily tasks, typically only access the User interface, not the Admin interface. (There are exceptions; some end-users have [permissions to also access the Admin interface](../../users-sources/access-control/manage_permissions.md#assign-can-access-admin-interface-permissions).

users-sources/access-control/manage_permissions/#assign-can-view-admin-interface-permissions), and conversely, some end-users never go to the User interface, but rather [log directly into their application](/brands/#external-user-settings) and log in using authentik behind the scenes.)

:::info
This document covers the basic tasks that end-users accomplish in the User interface. All of our technical documentation is available to all users, just note that the vast majority of it is for the Admin interface because the User interface is for simple end-user tasks.
:::

## User details

This section of the User interface displays (and allows you to edit) the following fields:

- **Username**: the username is the unique identifier associated with the user, and is required for logging in.
- **Name**: a display name, or nickname, for the user.
- **Email**: the email address for the user. This field can only be edited by the user if they have permissions, which are configured globally in the [System settings](../..//sys-mgmt/settings.md#allow-users-to-change-email) or [per individual user](../user/user_ref.mdx#goauthentikiousercan-change-email) by setting it as a flag (`can-change-email: true`) in the **Attributes** field for the user.
- **Locale**: here you can override any global locale settings and either choose a specific language or select **Auto-Detect**, which relies on the user's browser local settings.
- **Change your password**: If a user has the permission, they can modify their password here.

## Sessions

This tab shows all current sessions that are running for the user. Here you can delete sessions, including the current one (which would result in an automatic log out) or a session on a remote device.

## Consent

Here you can view applications that are configured to use the Consent stage, which administrators can add to an authorization flow.

When an administrator adds this stage to an authorization flow, the user logging in is presented with a pop-up confirmation page asking if they agree to allow the application (the relying party or RP) to directly request their account data (typically profile and email address) from the OAuth source. The user clicks **Continue** to give consent.

For more information refer to our documentarton on the [Consent stage](../../add-secure-apps/flows-stages/stages/consent/index.md).
