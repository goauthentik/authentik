## Highlights

## Breaking changes

The `hash_password` management command no longer accepts a password as a positional command-line argument. Run the command without arguments to enter the password at a hidden interactive prompt:

```bash
docker compose run --rm server hash_password
```

For automation, pipe the password through standard input:

```bash
echo "$PASSWORD" | docker compose run --rm server hash_password
```

By default, imported password hashes must use `pbkdf2_sha256`, `bcrypt_sha256`, `scrypt`, or `argon2`. The configured [work factors](https://docs.djangoproject.com/en/stable/topics/auth/passwords/#increasing-the-work-factor) and [salt entropy](https://docs.djangoproject.com/en/stable/topics/auth/passwords/#increasing-the-salt-entropy) must meet authentik's current security requirements.

This applies to `AUTHENTIK_BOOTSTRAP_PASSWORD_HASH`, blueprint `password_hash` values, and the `/api/v3/core/users/<id>/set_password_hash/` endpoint. API clients can set `override` to `true` to import another valid Django password hash after accepting the security and user-enumeration risks. Bootstrap variables and blueprints do not support this override.

## New features and improvements

### Base URL system setting

authentik now has a **Base URL** [system setting](../../sys-mgmt/settings.md) that records the external URL under which your instance is reachable, for example `https://authentik.company`. You can configure it in the Admin interface under **System** > **Settings** or through the API, and new installations are prompted for it during initial setup. To seed it at install time, use the new [`AUTHENTIK_WEB__BASE_URL`](../../install-config/configuration/configuration.mdx#authentik_web__base_url) configuration option; a value set through the Admin interface or API always takes precedence over it.

When you configure the Base URL, provide the scheme and host only. All of authentik is served under [`AUTHENTIK_WEB__PATH`](../../install-config/configuration/configuration.mdx#authentik_web__path) (`/` by default), and that path is applied on top of the Base URL. So even when you serve authentik under a subpath, for example `AUTHENTIK_WEB__PATH=/authentik/`, the Base URL stays `https://authentik.company` and must not include the `/authentik` prefix; the two are combined into links under `https://authentik.company/authentik/`.

This setting is currently unused and will become required starting with authentik 2026.11, so we recommend configuring it now.

## Upgrading

This release does not introduce any new requirements. You can follow the upgrade instructions below; for more detailed information about upgrading authentik, refer to our [Upgrade documentation](../../install-config/upgrade.mdx).

:::warning
When you upgrade, be aware that the version of the authentik instance and of any outposts must be the same. We recommend that you always upgrade any outposts at the same time you upgrade your authentik instance.
:::

### Docker Compose

To upgrade, download the new docker-compose file and update the Docker stack with the new version, using these commands:

```shell
wget -O docker-compose.yml https://goauthentik.io/version/xxxx.x/lifecycle/container/compose.yml
docker compose up -d
```

The `-O` flag retains the downloaded file's name, overwriting any existing local file with the same name.

### Kubernetes

Upgrade the Helm Chart to the new version, using the following commands:

```shell
helm repo update
helm upgrade authentik authentik/authentik -f values.yaml --version ^xxxx.x
```
