How the Light DOM Migration Interacts with CSS Ownership

Right now you have a three-layer shadow DOM chain:

FlowExecutor (shadow) → stage component (shadow) → ak-flow-card (shadow)

Each shadow boundary requires its own copy of any PF CSS it references. That's
 why every stage imports PFLogin — the classes like .pf-c-login__main-body
used by ak-flow-card are trapped behind shadow boundaries.

When you move renderChallenge() output to light DOM, the chain collapses:

FlowExecutor (shadow)
  └─ slot → light DOM content styled by global CSS

Stage content in the light DOM gets styled by static.global.css, which already
 imports everything needed: PFLogin, PFButton, PFTitle, your login.css
override, and even FlowExecutor.css. That file was built for exactly this —
the comment at the top of your login.css says as much ("ShadyDOM compatibility
 mode").

Does the Vendoring Plan Create Complications?

No, but there's one structural constraint to respect.

The vendored layout files must remain importable by static.global.css. Right
now static.global.css does:

@import "./components/Login/login.css";
@import "#flow/FlowExecutor.css";

If you move these to src/flow/styles/, static.global.css needs to update its
import paths. That's trivial. The important thing is that the vendored CSS
serves two consumers:

1. FlowExecutor's shadow DOM static styles array (for the shadow-DOM-active
path)
2. static.global.css (for the light DOM / compatibility / future progressive
path)

As long as the vendored files remain plain CSS importable by both paths, no
complication.

What the Migration Actually Simplifies

The vendoring plan is aligned with the light DOM direction in a meaningful
way:

- FlowExecutor's shadow DOM shrinks to pure layout. The grid, the
locale-select positioning, the layout variants. That's it. No button styles,
no title styles, no list styles — all confirmed dead weight already. When
stages move to light DOM, even the card structure CSS (pf-c-login__main-body
etc.) leaves the shadow DOM and lives only in global CSS.
- Stages stop importing PFLogin entirely. Today each stage imports PFLogin
because ak-flow-card uses those classes behind its shadow boundary. Once card
content is in light DOM, static.global.css handles it. The per-stage static 
styles arrays shrink to just component-specific overrides.
- static.global.css becomes the single CSS authority for form-level styling.
It already is for the ShadyDOM path. The progressive migration just makes that
 the primary path instead of the fallback.

One Thing to Watch

Your login.css override has these selectors targeting content that currently
lives inside the shadow DOM:

.pf-c-login__main-footer .pf-c-button__icon { ... }
.pf-c-login__main-footer .source-button { ... }

These are descendant selectors that cross what will eventually be a
shadow/light DOM boundary shift. Today they work because they're imported into
 the shadow DOM. Post-migration they'd work because they're in
static.global.css. Both paths are covered, so no issue — but only if you don't
 accidentally remove these from static.global.css when vendoring.

Summary

The vendoring plan and the light DOM migration are complementary:

- Vendor the layout CSS into FlowExecutor's shadow DOM → that's the one layer
that stays shadow
- Let static.global.css remain the authority for everything below layout
(cards, forms, buttons, titles) → that's where light DOM content gets styled
- The 5 dead PF imports in FlowExecutor are dead now and become even more
irrelevant post-migration
- The 20+ stages importing PFLogin become progressively unnecessary as their
content moves to light DOM

The only risk would be if you vendored the CSS and then broke the
static.global.css import chain. Keep those paths working and you're clean.

Where Your Instinct Leads
