between title/body/footer areas)
- Hide/show transitions between stages

Stages:
- Form layout (field stacking, label alignment, spacing)
- Conditional visibility (show/hide authenticator options, toggle password
visibility)
- Input group composition (password field + reveal button as a unit)
- Stage-specific layout (TOTP QR code beside input, WebAuthn prompt centering)

These are all content layout — how the form elements relate to each other
inside the stage area. Distinct from page layout (where the stage area sits on
 the viewport).

The Light DOM Complication

If inputs, selects, and forms live in the light DOM, the executor needs to
provide default CSS for them. A white-label reseller who replaces the
page/frame but keeps the executor will get:

- A stage that produces form elements into the light DOM
- Those elements need some styling to be usable out of the box
- The reseller can override that CSS, but a bare executor without any form
styles would be unusable

So the executor's CSS deliverable is something like:

┌───────────┬────────────────────────────┬───────────────────────────────┐
│   Layer   │         CSS scope          │   Replaceable by reseller?    │
├───────────┼────────────────────────────┼───────────────────────────────┤
│ Page      │ Grid, background, floating │ Yes — replace entirely        │
│           │  controls, layout variants │                               │
├───────────┼────────────────────────────┼───────────────────────────────┤
│ Frame     │ Card appearance, logo,     │ Yes — replace entirely        │
│           │ brand footer               │                               │
├───────────┼────────────────────────────┼───────────────────────────────┤
│           │ Form layout, input         │ Override-friendly defaults —  │
│ Executor  │ styling, field spacing,    │ reseller extends or           │
│ defaults  │ hide/show, stage-specific  │ overrides, doesn't replace    │
│           │ arrangements               │ wholesale                     │
└───────────┴────────────────────────────┴───────────────────────────────┘

The executor defaults are the baseline that makes the login functional without
 any page or frame. Ugly, unstyled in terms of brand, but the form works,
fields are stacked sensibly, buttons are clickable, conditional sections show
and hide correctly.

What This Means for the CSS Build

flows.global.css splits into (conceptually, not necessarily literally at this
stage):

1. Page/Frame CSS — the grid, branding, layout variants. This is what
authentik provides as its default look. Resellers swap this.
2. Executor/Stage defaults — form layout, input groups, field spacing,
visibility control. This ships with the executor. Resellers can override via
specificity or custom properties, but they probably keep most of it.

The current mess is that these two are interleaved — .pf-c-login__main-body
padding is page-frame CSS applied to stage content, .pf-c-login__main-footer
is frame CSS that also controls stage action-link layout. PFLogin is imported
everywhere because it contains rules for both layers with no separation.

When you vendor, this is the real organizing principle: not "which PF file did
 this come from" but "is this page/frame or executor/stage." The PF origin
becomes irrelevant — what matters is which layer owns the rule and whether a
reseller should be able to replace it independently.

1. The Global CSS (flows.global.css) and the HTML Template

The global CSS needs to contain:

- Font faces — RedHat font, icon fonts (FA, PF icons). These are pure assets,
